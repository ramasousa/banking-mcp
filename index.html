A tese

Não é sobre chatbot.
É sobre distribuição.

APIs bancárias já existem. Agentes de IA já existem. O MCP Server é o elo que falta — o protocolo padrão que permite qualquer modelo acessar qualquer produto Bradesco de forma governada, auditada e regulatoriamente correta.

01
O Axway já tem tudo
4.400+ APIs, 24 gateways, 23 políticas de segurança, 30 bilhões de requisições por mês. O Bradesco não precisa construir nada novo no backend — precisa expor o que já existe para um novo canal: os agentes de IA.
02
MCP é o protocolo que faltava
Model Context Protocol (MCP) é o padrão aberto que define como um agente de IA chama uma ferramenta externa. Claude, GPT e Gemini todos entendem MCP — ou têm adaptadores equivalentes. Uma especificação de tool, todos os modelos.
03
Governança na camada certa
O Gate de Política, a validação de token do usuário, o LgpdSanitizer, o audit log — tudo fica no MCP Server. O modelo de IA nunca toca dados bancários diretamente. Governança determinística, não delegada ao modelo.
04
Dois cases, um padrão
VaaS MCP Server: operações (comprar veículo, pagar tributos, contratar seguro). Banking MCP Server: dados pessoais (saldo, extrato, fatura, apólice). O mesmo padrão arquitetural cobre ambos — só muda o tipo de autenticação e o escopo dos dados.

Pleito formal de integração com a Bridge
O VaaS foi desenvolvido usando Amazon Bedrock AgentCore como prova de conceito. A proposta é migrar a camada de orquestração para a Bridge — plataforma de IA do Bradesco — e usar o MCP Server como a interface de tools. Isso transforma o VaaS no primeiro produto nativo Bridge da Tribe Open Platform.

O que estamos pedindo: acesso às APIs de Bradesco Seguros, CaaS, LaaS, PIX, Arrecadação e BaaS Core em ambiente de homologação — para execução da POC em 8 semanas.
