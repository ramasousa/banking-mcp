Visão Geral — O que é e o que faz

O MCP Server VaaS é a camada que transforma APIs existentes publicadas no Axway em ferramentas inteligentes consumíveis pela Bridge. Ele não reimplementa nenhuma lógica de negócio — ele expõe, governa e enriquece semanticamente chamadas que já existem.

Responsabilidades do MCP Server — e o que ele explicitamente não faz

Faz
Expõe tools via protocolo MCP — responde a tools/list e tools/call seguindo o padrão MCP oficial

Autentica no Axway — gerencia tokens OAuth2 com refresh automático

Mapeia tool → API Axway — cada tool tem um mapeamento explícito para método, path e API ID no Axway

Enriquece respostas semanticamente — adiciona contexto útil para o agente (ex: calculado automaticamente, incentivo aplicado, prazo estimado)

Aplica middleware — gate de política, idempotência, auditoria, circuit breaker — antes e depois de cada tool call
Não faz
Não reimplementa regras de negócio — a lógica de crédito está no CaaS, a atuarial está no Seguros. O MCP Server é um proxy inteligente, não um novo domínio

Não decide o que executar — isso é responsabilidade do agente na Bridge. O MCP Server executa o que foi solicitado (após o middleware aprovar)

Não acessa backends diretamente — toda chamada passa pelo Axway. Nunca contorna o gateway

Não armazena dados de negócio — persiste apenas chaves de idempotência (DynamoDB) e logs de auditoria (CloudTrail/S3)
Princípio fundamental: o MCP Server é um tradutor de intenção para chamada de API — com governança. A Bridge diz "quero calcular tributos para MG", o MCP Server transforma isso em GET /veiculos/tributos/v1?uf=MG&fipe=139900 autenticado no Axway, e devolve uma resposta enriquecida que o agente consegue raciocinar.

Entry point — como o servidor inicia

// src/server.ts
import { McpServer, StreamableHTTPServerTransport } from '@modelcontextprotocol/sdk/server';
import { ToolRegistry } from './tools/registry';
import { AxwayClient } from './axway/client';
import { MiddlewarePipeline } from './middleware/pipeline';

async function main() {
  // 1. Inicializa o cliente Axway (faz o primeiro token fetch)
  const axway = await AxwayClient.init({
    baseUrl:      process.env.AXWAY_BASE_URL,
    clientId:     process.env.AXWAY_CLIENT_ID,     // Secrets Manager
    clientSecret: process.env.AXWAY_CLIENT_SECRET,  // Secrets Manager
    scopes:       ['veiculo:read', 'credito:simulate', 'seguro:quote',
                   'pagamento:write', 'arrecadacao:write']
  });

  // 2. Cria o servidor MCP
  const server = new McpServer({
    name:    'vaas-mcp-server',
    version: '1.0.0'
  });

  // 3. Registra todas as tools
  const pipeline = new MiddlewarePipeline();
  await ToolRegistry.registerAll(server, axway, pipeline);

  // 4. Expõe via HTTP/SSE — compatível com Bridge e Bedrock AgentCore
  const transport = new StreamableHTTPServerTransport({ port: 3000 });
  await server.connect(transport);

  console.log('vaas-mcp-server pronto na porta 3000');
}

main().catch(console.error);

Como uma tool é definida — tipo base

// src/tools/types.ts
export interface ToolDefinition<TInput = unknown, TOutput = unknown> {
  name:        string;           // identificador único — snake_case
  description: string;           // para o agente entender QUANDO chamar esta tool
  inputSchema: object;           // JSON Schema — validado antes de chamar o Axway
  allowsWrite: boolean;          // true = passa pelo Gate de Política
  idempotent:  boolean;          // true = verifica DynamoDB antes de executar
  axwayMap:    AxwayMapping;      // mapeamento para o Axway
  enrich?:     (raw: TOutput, input: TInput) => TOutput; // enriquecimento semântico opcional
}

export interface AxwayMapping {
  apiId:   string;    // ID da API no Axway API Manager
  method:  string;    // HTTP method
  path:    string;    // path com parâmetros em {chave}
  scopes:  string[];  // OAuth2 scopes necessários
  timeout: number;    // ms — cada API tem seu SLA
}
03

Integração com o Axway

O MCP Server é um consumer OAuth2 das APIs do Axway. Toda chamada passa pelo gateway — nunca diretamente ao backend. Aqui estão todos os detalhes do fluxo de autenticação e do cliente HTTP.

Fluxo de autenticação — OAuth2 Client Credentials

// src/axway/auth.ts
export class AxwayTokenManager {
  private cache: { token: string; expiresAt: number } | null = null;

  async getToken(): Promise<string> {
    // Renova 60s antes de expirar — nunca serve token vencido
    if (this.cache && Date.now() < this.cache.expiresAt - 60_000) {
      return this.cache.token;
    }

    const response = await fetch(`${this.config.baseUrl}/oauth/token`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: new URLSearchParams({
        grant_type:    'client_credentials',
        client_id:     this.config.clientId,
        client_secret: this.config.clientSecret,
        scope:         this.config.scopes.join(' ')
      })
    });

    if (!response.ok) throw new Error(`Axway auth falhou: ${response.status}`);

    const data = await response.json();
    this.cache = {
      token:     data.access_token,
      expiresAt: Date.now() + (data.expires_in * 1000)
    };
    return this.cache.token;
  }
}

Cliente HTTP — correlação, retry e timeout

// src/axway/client.ts
export class AxwayClient {
  async call<T>(
    mapping: AxwayMapping,
    params:  Record<string, unknown>,
    context: ToolContext
  ): Promise<T> {
    const token = await this.tokenManager.getToken();
    const url   = this.buildUrl(mapping.path, params);

    // Cabeçalhos obrigatórios em TODA chamada ao Axway
    const headers = {
      'Authorization':     `Bearer ${token}`,
      'X-Correlation-ID':  context.correlationId,  // sagaId para rastreabilidade
      'X-MCP-Tool-Name':   context.toolName,        // qual tool chamou
      'X-Agent-Session-ID': context.agentSessionId, // sessão do agente na Bridge
      'Content-Type':      'application/json'
    };

    // Retry com backoff exponencial — apenas para 5xx e timeout
    return await this.withRetry(async () => {
      const res = await fetch(url, {
        method:  mapping.method,
        headers,
        body:    mapping.method !== 'GET' ? JSON.stringify(params) : undefined,
        signal:  AbortSignal.timeout(mapping.timeout)
      });

      if (res.status === 401) { // token expirou de forma inesperada
        this.tokenManager.invalidate();
        throw new RetryableError('Token expirado — renovando');
      }
      if (res.status === 429) { // rate limit do Axway
        const retryAfter = res.headers.get('Retry-After') ?? '5';
        throw new RateLimitError(parseInt(retryAfter) * 1000);
      }
      if (!res.ok) throw new AxwayApiError(res.status, await res.json());

      return res.json() as Promise<T>;
    }, { maxAttempts: 3, baseDelayMs: 500 });
  }
}

Considerações críticas sobre o Axway

Autenticação no Axway
O MCP Server precisa de um client OAuth2 registrado no Axway API Manager — identidade de serviço (não humana). A senha do client deve estar no AWS Secrets Manager, nunca em variável de ambiente direta.

O client precisa ter subscriptions ativas nas APIs que quer chamar. Sem subscription, o Axway retorna 403 mesmo com token válido.
Rate limits herdados
O Axway já tem 500 TPS por verbo HTTP nas políticas existentes. O MCP Server não contorna isso — o circuit breaker interno age antes de atingir o limite do Axway, protegendo os backends.

Para a POC VaaS, o volume de calls de ferramenta do agente é muito abaixo desse limite — mas é importante medir e expor a métrica vaas.axway.ratelimit.hits.
Correlation ID
O cabeçalho X-Correlation-ID deve carregar o sagaId da transação VaaS em toda chamada ao Axway. Isso permite cruzar logs do Axway com logs do MCP Server e do agente — rastreabilidade end-to-end sem instrumentação adicional.
API ID vs. Path
O Axway pode ter múltiplas versões da mesma API (/v1, /v2). O mapeamento de tool deve referenciar o API ID do Axway, não só o path — isso garante que uma atualização de versão no gateway seja capturada sem mudar o código da tool.

Middleware Chain

Toda tool call passa pela mesma pipeline de middleware, na ordem definida. O middleware não conhece a tool específica — só o contexto e os metadados (allowsWrite, idempotent).

// src/middleware/pipeline.ts — executa em ordem para toda tool call
async function executePipeline(
  toolDef: ToolDefinition,
  input:   unknown,
  ctx:     ToolContext,
  handler: () => Promise<unknown>
): Promise<unknown> {

  // ① Trace — abre X-Ray segment
  return await tracer.trace(ctx.correlationId, async () => {

    // ② Validação de schema — antes de qualquer lógica
    SchemaValidator.validate(input, toolDef.inputSchema);

    // ③ Circuit breaker — verifica se a API destino está saudável
    CircuitBreaker.checkOrThrow(toolDef.axwayMap.apiId);

    // ④ Gate de Política — apenas para tools de escrita
    if (toolDef.allowsWrite) {
      await GatePolicyMiddleware.enforce(input, ctx);
    }

    // ⑤ Idempotência — apenas para tools de mutação idempotentes
    if (toolDef.idempotent) {
      const key = IdempotencyMiddleware.buildKey(toolDef.name, input, ctx);
      if (!await IdempotencyMiddleware.tryAcquire(key)) {
        return await IdempotencyMiddleware.getCached(key); // retorna resultado anterior
      }
    }

    // ⑥ Executa a chamada real ao Axway
    const result = await handler();

    // ⑦ Enriquecimento semântico — adiciona contexto para o agente
    const enriched = toolDef.enrich ? toolDef.enrich(result, input) : result;

    // ⑧ Auditoria — grava no CloudTrail (nunca bloqueia a resposta)
    AuditMiddleware.record(toolDef.name, input, enriched, ctx).catch(console.error);

    return enriched;
  });
}
①
X-Ray TraceAbre um subsegment do X-Ray com o nome da tool e o correlationId. Toda a chain aparece como um único trace rastreável no console AWS.
②
Schema ValidatorValida o input contra o JSON Schema da tool antes de qualquer chamada de rede. Erros de validação retornam imediatamente ao agente sem consumir quota do Axway.
③
Circuit BreakerVerifica o estado do circuit breaker por API ID do Axway. Se o breaker estiver OPEN, retorna erro imediatamente com mensagem legível para o agente — não tenta a chamada.
④
Gate de PolíticaPara tools com allowsWrite: true: chama verificar_gate_politica deterministicamente. Se não aprovado, lança GatePolicyError — o agente recebe a mensagem de bloqueio e decide como proceder.
⑤
Idempotência DynamoDBPara tools com idempotent: true: tenta gravar a chave com attribute_not_exists(pk). Se a chave já existe, retorna o resultado da execução anterior — o agente recebe a mesma resposta de uma execução bem-sucedida.
⑥
Axway ClientExecuta a chamada HTTP real com o token OAuth2, cabeçalhos de correlação e retry automático para erros 5xx e timeouts.
⑦
Response EnricherTransforma a resposta raw do Axway adicionando campos calculados, contexto de negócio e linguagem natural que ajuda o agente a raciocinar sobre o resultado.
⑧
Audit LoggerGrava de forma assíncrona (non-blocking) no CloudTrail: nome da tool, input sanitizado (PII removido), output hash, correlationId, timestamp. Nunca atrasa a resposta ao agente.

Segurança

Quatro camadas de segurança independentes — cada uma falha de forma segura mesmo que a anterior falhe.

Autenticação de entrada
A Bridge autentica no MCP Server
O MCP Server exige um Bearer token da Bridge em toda requisição de entrada — não aceita chamadas anônimas. Token validado via JWKS do IAM da Bridge (ou mTLS se disponível). Sem token válido: HTTP 401 antes de qualquer tool ser processada.
Scoping de tools
O agente só vê o que pode usar
O tools/list retorna apenas as tools habilitadas para o contexto da chamada. O ambiente HML não expõe tools de produção. Uma tool de pagamento em HML aponta para o simulador Axway — sem risco de acionar pagamento real.
PII fora do agente
Dados sensíveis ficam no MCP Server
O agente nunca recebe dados bancários reais. O campo beneficiario_token é um token opaco — o MCP Server resolve para os dados reais na hora da chamada ao Axway. CPF, conta bancária e chave PIX nunca aparecem no reasoning do agente.
Vault de credenciais
Zero credenciais em código ou env
AXWAY_CLIENT_ID e AXWAY_CLIENT_SECRET são lidos do AWS Secrets Manager no startup. Rotação automática a cada 30 dias. A IAM Task Role do ECS tem acesso apenas ao secret específico do vaas-mcp-server.

Sanitização de PII no Audit Log

// src/middleware/audit.ts — remove PII antes de logar
const PII_FIELDS = ['cpf', 'email', 'telefone', 'conta', 'agencia', 'chave_pix'];

function sanitize(obj: unknown): unknown {
  if (typeof obj !== 'object' || obj === null) return obj;
  return Object.fromEntries(
    Object.entries(obj as object).map(([k, v]) => [
      k,
      PII_FIELDS.includes(k) ? '[REDACTED]' : sanitize(v)
    ])
  );
}

Architecture Decision Records

Decisões tomadas com alternativas consideradas e raciocínio explícito.

MCP-ADR-001
Runtime: TypeScript/Node.js em vez de Java Spring Boot
Aprovado
Contexto:O stack Java Spring Boot já existe para os domain services. O MCP Server é um componente novo — thin proxy, sem lógica de domínio.
Decisão:TypeScript com o SDK oficial @modelcontextprotocol/sdk. O SDK MCP é TypeScript-first — a implementação Java é community-maintained e menos madura.
Alternativas:Java Spring Boot (SDK menos maduro, cold start mais lento para um proxy thin) · Python (também tem SDK oficial mas menos familiaridade no time)
Consequência:O time precisa ter desenvolvedores confortáveis com TypeScript para manutenção. A Stack Java continua nos domain services — o MCP Server é um componente complementar.
MCP-ADR-002
Tools manuais para a POC, não auto-descoberta do Axway Catalog
Aprovado
Contexto:O Axway tem 4.400+ APIs. Poderia se auto-descobrir tools do catálogo via API do Axway Manager.
Decisão:8 tools definidas manualmente para a POC. Auto-descoberta é a visão de longo prazo — implementar na Fase 2.
Alternativas:Auto-descoberta agora: risco de expor APIs não adequadas ao agente, schema de input não otimizado para o agente, enricher genérico demais.
Consequência:Cada nova tool requer um PR. Para a escala futura (centenas de tools), a auto-descoberta precisa ser implementada com scoping rigoroso.
MCP-ADR-003
Gate de Política no MCP Server, não só no Domain Service
Aprovado
Contexto:O Gate de Política já existe no vaas-pagamento-service (Spring Boot). Deveria ser reutilizado ou reimplementado?
Decisão:O MCP Server chama o Gate via tool verificar_gate_politica (que é também ela mesma uma tool exposta ao agente). A fonte da verdade é única — o domain service Java. O MCP Server não reimplementa.
Consequência:Adiciona uma chamada de rede extra antes de cada write. Aceitável dado que o Gate retorna em <1s e o benefício de ter uma única fonte de verdade para a política compensa.
MCP-ADR-004
Transport HTTP/SSE — não stdio, não WebSocket
Aprovado
Contexto:MCP suporta três transports: stdio (processo local), HTTP/SSE, WebSocket.
Decisão:HTTP com Server-Sent Events (SSE) via StreamableHTTPServerTransport. Padrão recomendado para servidores remotos — compatível com Bedrock AgentCore e Bridge.
Consequência:O servidor expõe porta 3000. O ALB termina TLS e faz proxy para a task ECS. A Bridge/AgentCore conecta via HTTPS — o SSE fica sobre o mesmo canal.
