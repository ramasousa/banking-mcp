01

O que muda — e o que reaproveita

A resposta direta: a estrutura reaproveita. Mas dois pontos mudam de forma fundamental e precisam ser resolvidos antes de qualquer implementação.

Muda — crítico
Modelo de autenticação
O VaaS usa client_credentials — identidade de serviço. Saldo, extrato e fatura são dados pessoais do usuário — exigem que o próprio usuário autorize o acesso.

Isso muda o fluxo OAuth2 para Authorization Code + PKCE. O token que chega ao Axway carrega o sub (identidade) do usuário — não do serviço.

Sem isso: o Axway entregaria dados de um cliente para outro. Isso é uma brecha de sigilo bancário — não um bug de software.
Muda — técnico
Protocolo de comunicação
Claude fala MCP (protocolo nativo Anthropic). GPT fala Function Calling (formato OpenAI). Gemini fala Function Declarations (formato Google).

São três formatos diferentes para a mesma intenção — "execute esta função com estes parâmetros". É preciso uma camada de tradução de protocolo que converte cada formato para o executor central compartilhado.

Sem isso: seria necessário implementar e manter três servidores separados com a mesma lógica.
Reaproveita — sem alteração
Estrutura do executor
ToolDefinition, AxwayMapping, MiddlewarePipeline, AxwayClient, CircuitBreaker, AuditMiddleware — idênticos ao VaaS MCP Server.

Deployment (ECS Fargate), observabilidade (CloudWatch + X-Ray), secrets (Secrets Manager) — idênticos.
Reaproveita — com adaptação
Gate e Idempotência
Tools de consulta (saldo, extrato) são leitura — não precisam de Gate de Política.

Idempotência continua relevante para consultas em cache: a mesma pergunta "qual meu saldo?" não deve gerar duas chamadas Axway em 30 segundos. Adaptar o IdempotencyMiddleware para suportar TTL curto de cache de leitura.

03

Autenticação do Usuário — A Mudança Crítica

Este é o ponto mais importante da especificação. Dados bancários pessoais exigem que o usuário consinta e que o token carregue a identidade dele — não do serviço.

Por que não usar client_credentials aqui: com client_credentials, o MCP Server tem um token de serviço que pode acessar qualquer conta bancária do Axway que o serviço tenha permissão. Uma falha de lógica no mapeamento de usuário expõe dados de outros clientes. Com Authorization Code, o token tem o sub do usuário embutido — o Axway rejeita na camada de gateway qualquer tentativa de acessar dados de outro usuário, independente do que o serviço solicitar.

Fluxo completo — do app ao dado bancário

1
Usuário já está autenticado no app BradescoO app Bradesco (mobile ou web) já completou o Authorization Code + PKCE flow com o servidor de identidade Bradesco (CIAM — Azure Entra ID ou Keycloak). O app tem um access_token com scopes do usuário.
2
App passa o token do usuário para o agente de IAQuando o usuário interage com o agente (Claude, GPT ou Gemini) dentro do app, o app inclui o token de usuário no contexto da requisição para o agente. O agente não conhece o token diretamente — ele chega ao Banking MCP Server via cabeçalho de autorização delegada.
3
Banking MCP Server valida o token de usuárioAntes de qualquer tool call, o UserAuthMiddleware valida o JWT do usuário via JWKS do CIAM Bradesco. Extrai o sub (userId), os scopes disponíveis e o exp (expiração). Rejeita com 401 se token inválido ou expirado.
4
Verifica se o scope cobre a tool solicitadaCada tool requer um scope específico. consultar_saldo exige conta:saldo:read. Se o token do usuário não tem esse scope (usuário não consentiu), retorna 403 com mensagem clara ao agente.
5
Chama o Axway com o token do usuário (não do serviço)O AxwayClient usa o token do usuário como Bearer — não o token de serviço do VaaS. O Axway valida que o sub do token corresponde ao cliente da conta solicitada. Blindagem nativa no gateway.
6
Token de serviço para auditoria complementarO Banking MCP Server ainda usa suas próprias credenciais de serviço para registrar a chamada no audit log — mas elas nunca chegam ao Axway como auth de acesso a dados. Separação de concerns: serviço autentica para logar, usuário autentica para acessar.

Implementação do UserAuthMiddleware

// src/middleware/user-auth.ts
import { createRemoteJWKSet, jwtVerify } from 'jose';

const JWKS = createRemoteJWKSet(
  new URL(process.env.CIAM_JWKS_URL) // ex: https://login.bradesco.com.br/.well-known/jwks.json
);

export async function validateUserToken(
  bearerToken: string,
  requiredScope: string
): Promise<UserContext> {

  // 1. Verifica assinatura, issuer e expiração
  const { payload } = await jwtVerify(bearerToken, JWKS, {
    issuer:   process.env.CIAM_ISSUER,   // ex: https://login.bradesco.com.br
    audience: 'bradesco-banking-mcp'
  });

  // 2. Verifica se o escopo necessário está presente
  const tokenScopes = (payload.scope as string || '').split(' ');
  if (!tokenScopes.includes(requiredScope)) {
    throw new ScopeError(`Token não possui o scope '${requiredScope}' necessário para esta consulta`);
  }

  // 3. Retorna contexto do usuário para uso no Axway call e auditoria
  return {
    userId:    payload.sub!,            // identidade do usuário — nunca exposta ao agente
    clienteId: payload.cliente_id as string, // ID interno Bradesco
    scopes:    tokenScopes,
    expiresAt: payload.exp! * 1000
  };
}
04

Tradução de Protocolo — Um Executor, Três Entradas

Cada modelo de IA usa um formato diferente para declarar e chamar tools. O servidor expõe três endpoints — um por modelo — que traduzem para o mesmo executor central.

Claude · MCP Nativo
Endpoint: /mcp (SSE)
SDK: @modelcontextprotocol/sdk
Protocolo: JSON-RPC 2.0

Claude entende MCP nativamente — sem adaptação de protocolo. O StreamableHTTPServerTransport já lida com o formato correto.
GPT · Function Calling
Endpoint: /openai/tools (REST)
SDK: openai npm package
Protocolo: OpenAI tools format

GPT declara tools como objetos { type: "function", function: {...} } e chama via tool_calls na resposta do chat. O adaptador converte para o executor interno.
Gemini · Function Declarations
Endpoint: /gemini/tools (REST)
SDK: @google/generative-ai
Protocolo: Gemini FunctionDeclaration

Gemini usa FunctionDeclaration com schema de tipo OBJECT diferente do JSON Schema padrão. O adaptador converte o schema e as chamadas.

O mesmo schema de tool em três formatos

Implementação da camada de adaptadores

// src/adapters/openai-adapter.ts — endpoint /openai/tools
import express from 'express';
const router = express.Router();

// GET /openai/tools — retorna lista de tools no formato OpenAI
router.get('/tools', (req, res) => {
  const openAiTools = ToolRegistry.getAll().map(tool => ({
    type: 'function',
    function: {
      name:        tool.name,
      description: tool.description,
      parameters:  tool.inputSchema  // JSON Schema é compatível diretamente
    }
  }));
  res.json({ tools: openAiTools });
});

// POST /openai/tools/call — recebe tool_call do GPT e executa
router.post('/tools/call', async (req, res) => {
  const { tool_call, user_token } = req.body;   // user_token vem do app
  const result = await CoreExecutor.execute({
    toolName:  tool_call.function.name,
    input:     JSON.parse(tool_call.function.arguments),
    userToken: user_token,
    source:    'gpt'
  });
  // Resposta no formato que o GPT espera como tool result
  res.json({ role: 'tool', tool_call_id: tool_call.id, content: JSON.stringify(result) });
});

// src/adapters/gemini-adapter.ts — endpoint /gemini/tools
router.post('/tools/call', async (req, res) => {
  const { functionCall, user_token } = req.body;
  const result = await CoreExecutor.execute({
    toolName:  functionCall.name,
    input:     functionCall.args,   // Gemini usa 'args' em vez de 'arguments'
    userToken: user_token,
    source:    'gemini'
  });
  res.json({ functionResponse: { name: functionCall.name, response: result } });
});

Catálogo de Tools Bancárias

4 tools de leitura. Nenhuma é de escrita — toda mutação (pagamentos, transferências) permanece fora do escopo desta versão e exige aprovação regulatória adicional.

consultar_saldo
LeituraCache 30s
Retorna o saldo disponível, bloqueado e limite de crédito da conta corrente ou poupança do usuário autenticado. Chamar sempre que o usuário perguntar sobre saldo, disponível ou limite. Nunca inferir saldo de operações anteriores.
input: tipo_conta?: "corrente" | "poupanca" // default: corrente output: saldo_disponivel: number // centavos — NUNCA float saldo_bloqueado: number // retido por TEDs, cheques, etc. limite_credito: number // apenas se conta com cheque especial data_atualizacao: string // ISO 8601 — agente deve informar ao usuário // NOTA: o userId vem do token — o agente NUNCA passa conta_id
Mapeamento Axway
GET /contas/v1/saldo · API ID: core-bancario-saldo-api · Scope: conta:saldo:read · Timeout: 3s
⚠ Auth obrigatória do usuário
O Axway extrai o sub do Bearer token e retorna apenas os dados da conta deste usuário. Sem token de usuário válido com scope conta:saldo:read → 403 no Axway antes de chegar ao backend.
consultar_extrato
LeituraCache 5min
Retorna as transações da conta corrente em um período. Usar para responder perguntas sobre movimentações, gastos por categoria, pagamentos realizados. O período máximo retornável é 90 dias — para períodos maiores, o usuário deve acessar o app diretamente.
input: periodo_dias: number // 7, 15, 30, 60 ou 90 — default: 30 tipo_lancamento?:"debito" | "credito" | "todos" // default: todos output: lancamentos: Array<{ data: string // ISO 8601 descricao: string // texto do lançamento (mascarado pelo enricher se PII) valor: number // centavos · positivo=crédito · negativo=débito tipo: "debito" | "credito" | "pix" | "ted" | "boleto" saldo_apos: number // saldo após este lançamento }> total_debitos: number // adicionado pelo enricher total_creditos: number // adicionado pelo enricher
Mapeamento Axway
GET /contas/v1/extrato?dias={periodo_dias}&tipo={tipo_lancamento} · API ID: core-bancario-extrato-api · Scope: conta:extrato:read · Timeout: 6s
⚠ Dado mais sensível — scope separado
Extrato é mais sensível que saldo — scope distinto (conta:extrato:read) exige consentimento explícito adicional no fluxo de autorização. Usuário pode ter consentido com saldo mas não com extrato.
consultar_fatura_cartao
LeituraCache 5min
Retorna a fatura atual ou uma fatura fechada do cartão de crédito Bradesco. Inclui valor total, mínimo, data de vencimento e lista de compras. Para cartões adicionais, retorna apenas os dados do titular autenticado.
input: mes_referencia?: string // "YYYY-MM" — default: fatura atual aberta cartao_id?: string // default: cartão principal — 4 últimos dígitos output: valor_total: number valor_minimo: number data_vencimento: string status: "aberta" | "fechada" | "paga" | "vencida" compras: Array<{ data: string comercio: string valor: number parcela?: string // ex: "2/6" }> dias_para_vencimento: number // adicionado pelo enricher alerta_vencimento: string | null // "Vence em 2 dias" se urgente
Mapeamento Axway
GET /cartoes/v2/{cartao_id}/fatura?mes={mes_referencia} · API ID: cartao-fatura-api · Scope: cartao:fatura:read · Timeout: 5s
⚠ Dados de compras — PII
A lista de compras pode expor hábitos de consumo — dado sensível sob LGPD. O LgpdSanitizer remove compras em estabelecimentos classificados como sensíveis (saúde, serviços íntimos) antes de enviar ao agente.
consultar_apolice_seguro
LeituraCache 30min
Retorna as apólices de seguro ativas do usuário. Para seguro veicular, inclui dados do veículo, coberturas, prêmio e vencimento. Usar quando o usuário perguntar sobre seu seguro, cobertura ou próximo vencimento.
input: tipo?: "auto" | "vida" | "residencial" // default: retorna todos output: apolices: Array<{ numero: string tipo: "auto" | "vida" | "residencial" status: "ativa" | "suspensa" | "cancelada" premio_mensal: number vencimento: string // data da apólice coberturas: string[] veiculo?: { // apenas para tipo "auto" modelo: string placa: string // mascarado: ABC-1*** ano: number } dias_para_vencimento: number // enricher }>
Mapeamento Axway
GET /seguros/v1/apolices?tipo={tipo} · API ID: bradesco-seguros-apolice-api · Scope: seguro:apolice:read · Timeout: 4s
⚠ Dado sensível
Apólice de seguro revela patrimônio e veículos do usuário. Cache mais longo (30min) é seguro porque apólices mudam raramente — mas o cache é por userId (nunca compartilhado entre usuários).

LGPD & Sigilo Bancário

Dados bancários pessoais têm dupla proteção: LGPD (dado pessoal financeiro) e Lei Complementar 105/2001 (sigilo bancário). O servidor precisa respeitar ambas.

O que o agente NUNCA deve receber: número completo da conta, CPF do usuário, número completo do cartão, chave PIX, dados de estabelecimentos classificados como sensíveis (farmácias, clínicas, etc. na fatura), endereço (da apólice residencial). O LgpdSanitizer remove esses campos antes de qualquer dado chegar ao contexto do modelo.

// src/middleware/lgpd-sanitizer.ts
const SENSITIVE_CATEGORIES = [
  'FARMAC', 'CLINICA', 'HOSPITAL', 'LABORAT', 'PSICOLOG',
  'UROLOGIS', 'GINECOL', 'ONCOLOG'
];

export function sanitizeForModel(toolName: string, rawResponse: unknown): unknown {
  switch (toolName) {
    case 'consultar_extrato':
      return sanitizeExtrato(rawResponse);
    case 'consultar_fatura_cartao':
      return sanitizeFatura(rawResponse);
    default:
      return rawResponse;
  }
}

function sanitizeFatura(fatura: any) {
  return {
    ...fatura,
    compras: fatura.compras.map((c: any) => ({
      ...c,
      comercio: isSensitiveCategory(c.comercio)
        ? 'Serviço de saúde'   // não revela qual clínica
        : c.comercio
    }))
  };
}

function isSensitiveCategory(nome: string): boolean {
  const upper = nome.toUpperCase();
  return SENSITIVE_CATEGORIES.some(cat => upper.includes(cat));
}
Base legal LGPD para esta integração: execução de contrato (Art. 7º, V) — o usuário tem contrato com o Bradesco e consultar seus próprios dados está dentro do objeto do contrato. O consentimento explícito (Art. 7º, I) é capturado no fluxo OAuth2 de autorização de scopes — cada scope é um consentimento distinto.
07

Deployment — Módulo separado, patterns compartilhados

Serviço separado do VaaS MCP Server por postura de segurança diferente, mas deploy em ECS Fargate com a mesma estrutura.

Por que serviço separado — não o mesmo container

IAM Task Role diferente
O Banking MCP Server precisa acessar o JWKS do CIAM para validar tokens de usuário. O VaaS MCP Server não precisa disso. Least privilege exige roles separadas.
Secrets separados
As credenciais Axway do Banking Server têm scopes de leitura de dados pessoais — muito mais restritivos. Vazamento de um não compromete o outro.
Escala independente
Consultas bancárias têm picos diferentes de pagamentos VaaS. ECS Auto Scaling opera por serviço — containers independentes escalam independentemente.

08

Architecture Decision Records

BANK-ADR-001
Authorization Code + PKCE — não client_credentials para dados pessoais
Contexto:O VaaS MCP Server usa client_credentials (identidade de serviço). O Banking Server poderia fazer o mesmo e usar o userId de outra fonte (sessão do app, header customizado).
Decisão:Usar o token OAuth2 do usuário como Bearer no Axway. O Axway valida que o sub do token é o dono da conta solicitada — isso é uma garantia da camada de gateway, não de código de aplicação.
Alternativa descartada:client_credentials + header X-User-ID: o Axway não validaria que o serviço tem permissão de acessar aquele usuário específico. Dependeria 100% da lógica do banking-mcp-server não ter bugs de autorização — risco inaceitável para dados bancários.
BANK-ADR-002
Um executor central com três adaptadores de protocolo
Contexto:Poderia criar três servidores independentes (um por modelo) ou um servidor com adaptadores.
Decisão:Um servidor, três endpoints de protocolo, um executor central. A lógica de tool, middleware e Axway client é implementada uma vez. Os adaptadores são thin — só traduzem o envelope do protocolo.
Consequência:Quando uma tool muda (ex: novo campo no extrato), a mudança reflete nos três modelos automaticamente. Quando um modelo lança uma nova versão do protocolo, só o adaptador daquele modelo precisa ser atualizado.
BANK-ADR-003
Cache por userId — não cache global
Contexto:O cache de leitura (saldo 30s, extrato 5min) precisa ser isolado por usuário.
Decisão:Chave do cache: hash(userId + toolName + inputHash). Nunca hash(toolName + inputHash). Um cache global exporia o saldo de um usuário para outro.
Implementação:ElastiCache Redis com TTL por entry. Em memória é insuficiente porque múltiplas instâncias ECS teriam caches independentes — um usuário poderia bater no Axway N vezes se fosse roteado para instâncias diferentes.
BANK-ADR-004
Escopo por tool — não escopo global de "leitura bancária"
Contexto:Poderíamos pedir um único scope bancario:read que cobre saldo, extrato, fatura e apólice.
Decisão:Scopes granulares: conta:saldo:read, conta:extrato:read, cartao:fatura:read, seguro:apolice:read. Princípio de mínimo privilégio e granularidade de consentimento.
Consequência:O usuário pode consentir com saldo mas não com extrato. Uma brecha de segurança no tratamento de extrato não expõe saldo — os scopes são validados independentemente no Axway.
